Flourisher CLI

Search results as a terminal product surface.

This repo starts with one hard-coded command, then documents how the same command can grow into a polished human workflow and a precise agent interface.

Copy commands Read decisions

What exists

flourisher search "analytics" returns the same 18 demo companies for every query. The table includes linked business names, linked profiles, Stripe integration notes, link acceptance, projects, verification level, products, and likely users.

What is deliberately fake

There is no live backend, ranking model, or customer data. The hard-coded dataset lets reviewers focus on output shape, ergonomics, and agent contracts before arguing about data quality.

Reproducible demo

Commands to paste

Default table

node ./bin/flourisher.js search "analytics"

Wide screenshot mode

node ./bin/flourisher.js search "analytics" --no-links --columns 160

Agent JSON

node ./bin/flourisher.js search "analytics" --output json --fields businessName,username,acceptsLink

Runtime contract

node ./bin/flourisher.js describe search

Profile detail

node ./bin/flourisher.js profile atlasmetrics

Compare rows

node ./bin/flourisher.js compare atlasmetrics vectorgrove summitschema

Decisions

What this version optimizes for

Human default

The default output is a table because the first workflow is scanning and comparing companies in a terminal.

Agent path

JSON, CSV, field masks, limits, and describe search avoid table scraping and reduce context usage.

Stable data shape

The display contract is explicit so a future backend can replace hard-coded rows without rewriting the CLI.

Responsive table

--wide, --compact, and --columns let demos and screenshots show how result density changes by terminal width.

Feature paths

Separate notes for follow-up work

Stubbed search table Human-focused terminal UI Responsive result density Agent-facing flags Backend contract Saved workflows Verification and PR flow

What else we could do

Build a full TUI with row navigation, tabs, detail panes, compare mode, saved notes, copy/open shortcuts, and progressive loading. Ink is the shortest path from this Node implementation; Bubble Tea is the strongest option if the project moves to Go.

What agents need

Keep adding structured outputs, schema introspection, field masks, input hardening, pagination, dry-run semantics for writes, and eventually an MCP surface. The CLI should be useful through both shell execution and typed tool calls.
